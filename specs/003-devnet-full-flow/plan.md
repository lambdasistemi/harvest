# Implementation Plan: Devnet End-to-End Full Protocol Flow

**Branch**: `003-devnet-full-flow` | **Date**: 2026-04-20 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/code/harvest-009/specs/003-devnet-full-flow/spec.md`

## Summary

Extend the single-spend devnet harness merged in issue #15 into a full
protocol-flow E2E suite covering every role in the Harvest constitution
(coalition, issuer, shop, reificator, customer) and every constitutional
lifecycle step (coalition create → shop onboard → topup → two
settlements → redemption → revert → reificator revocation). The scope is
a **prototype, not production**: the on-chain membership representation
is a plain set carried in a coalition-metadata UTxO datum and consumed
as a reference input — not an MPF and not MPFS-mediated. Every
constitutional invariant required for proof soundness (Principle V),
monotonic state (Principle VI), and the reification model (Principle
VII) is exercised exactly as it will be exercised under MPF/MPFS in
later tickets; only scale/usability differs.

**Technical approach**: reuse the merged `withDevnet` bracket,
`SpendSetup`, `SpendScenario`, and `Mutations` framework from
`DevnetSpendSpec`. Introduce a small Aiken module set for the
coalition-metadata validator, the per-customer voucher validator (a
natural extension of the merged `voucher_spend`), the redemption
validator, and the revert validator. On the off-chain side, add a
handful of pure Haskell action functions — one per transition — whose
signatures mirror the Lean state-machine transitions being scaffolded in
`/code/harvest-009/lean/` by a parallel agent. The scenarios are driven
by a thin `HarvestFlow` harness that composes those transitions and
asserts on the resulting UTxO set after each submission.

## Technical Context

**Language/Version**: Haskell GHC 9.10 (off-chain + tests), Aiken 1.1.x
(Plutus V3, on-chain), Node 20 (fixture producer — unchanged from #15).
Lean 4 (model twin) is authored in a parallel agent; this plan only
consumes its transition signatures.

**Primary Dependencies**:
- `cardano-node-clients` + `cardano-node-clients:devnet` — same merged
  surface used by #15 (`withDevnet`, `Submitter`, `Provider`,
  `TxBuild`, `Evaluate`). No new upstream patch anticipated.
- `hspec`, `aeson`, `base16-bytestring` — already present.
- Aiken on-chain: `aiken-lang/stdlib`. No `merkle-patricia-forestry`
  dependency — FR-013 forbids MPF here.

**Storage**: continues to use fixtures under
`offchain/test/fixtures/`. Adds per-customer fixture bundles (2
customers × up to 2 certificates each) regenerated by the same
`circuits/generate_fixtures.js` pipeline. Devnet runtime lives in the
`withDevnet` temp dir, ephemeral per test run.

**Testing**: `cabal test` via `checks.unit-tests` nix derivation,
inheriting the `cardano-node` binary wiring introduced by #15. One
hspec module per user story: `DevnetFullFlowSpec` (P1), `DevnetRedeemSpec`
(P2), `DevnetRevertSpec` (P3), `DevnetRevocationSpec` (P3). Each module
owns its own `withDevnet`/`beforeAll` lifecycle so any one can be
selected in isolation (SC-003).

**Target Platform**: x86_64-linux CI (nix-first), `cardano-node` version
**10.7.1** — user correction over the 10.7.0 listed in the spec’s
earlier drafts.

**Project Type**: additions to an existing multi-language project
(Circom + Aiken + Haskell + Rust FFI). No new top-level project, no new
executables; one new Aiken validator family and four new Haskell test
modules plus their shared harness.

**Performance Goals**: within 2× of the #15 `DevnetSpendSpec` runtime
(SC-002). Devnet startup (~15 s) amortised across the scenarios in each
spec file; per-tx budget identical to #15 (~5 s each). Budget envelope:
+ ~5 minutes of incremental CI time total.

**Constraints**:
- FR-013 forbids any MPF / MPFS dependency. On-chain membership checks
  are linear list scans over the coalition-metadata datum.
- FR-014 forbids multi-certificate spend paths.
- SC-005 forbids pinning ledger error strings — negative assertions
  use `shouldSatisfy isRejected`.
- Security invariants from constitution §V (proof binding of `d` and
  `commit_spent`, customer Ed25519 binding of `acceptor_pk` and
  `TxOutRef`, issuer cap-certificate binding, enforced reificator
  authorisation) **must hold** in the prototype exactly as in the MPF
  version.

**Scale/Scope**: N ∈ {1, 2, 3} for every enumerated set (shops,
reificators, customers, certificates). Deliberately outside any
realistic production envelope; scale is the concern of issues #5
(MPF on-chain) and #8 (MPFS mediation), not this ticket.

## Constitution Check

*GATE: checked before Phase 0 research and re-checked after Phase 1
design.*

| Principle | Status | Notes |
|-----------|--------|-------|
| I. Coalition, Not Silos | **PASS** | Plan registers shops and reificators as first-class on-chain entries in the coalition-metadata datum. |
| II. User Has No Wallet | **PASS** | Every settlement, redemption, and revert tx is submitted by the reificator or the shop, never by the customer. |
| III. Smart Contract as Trust Layer | **PASS** | Accept/reject comes from real node submission against compiled validator bytecode. No re-implemented validator in tests. |
| IV. Privacy by Default | **PASS** | On-chain data remains commitments (`commit_spent`) or enumerated public keys; no balances leak. Fixtures use hashed user secrets. |
| V. Proof Soundness | **PASS** | Every accepted path requires a valid Groth16 proof + customer Ed25519 signature binding `d`, `acceptor_pk`, and a live `TxOutRef`. Reificator authorisation is enforced by a linear scan over the coalition-metadata set — security-equivalent to MPF membership. |
| VI. Monotonic State | **PASS** | Settlement only increases the per-customer `commit_spent`; revert decreases it but is an authorised rollback, not a balance mutation; redemption clears the pending entry by consuming the script UTxO without a replacement. |
| VII. Reification Model | **PASS** | The three lifecycle states (committed → redeemed / reverted) map to the three tx families (settlement / redemption / revert) and the tests cover each path. Device-loss recovery (§VII) is covered by Story 4 (revocation) combined with Story 3 (revert). |
| VIII. Economic Model | **PASS** | Reificator funds fees from its own UTxO; no change vs. #15. |
| IX. Three-Trie State | **NEGOTIATED** | The constitution §IX describes three MPF tries. This ticket uses a set-valued coalition-metadata UTxO instead, covered by FR-013 and the spec’s "prototype, not production" clause. This is deliberately captured as a refinement obligation: when #5/#8 land, the set-based shape is replaced by MPF roots without changing any invariant asserted here. Tracked in Complexity Tracking. |
| X. Correct Before Optimized | **PASS** | This ticket is the canonical instance of the principle: N ∈ {1,2,3}, full semantics, no scale infrastructure. |
| XI. Nix-First | **PASS** | Everything runs under `just ci`; no new non-nix dependency. |

**Re-check after Phase 1 design**: `data-model.md` and `contracts/`
preserve the table above verbatim. Principle IX remains negotiated in
the same way; no new violations introduced during design.

**Structure Decision**: strictly additive to the tree established in
#15. The only modifications to existing files are (a) `voucher_spend.ak`
to enforce reificator-set membership against the new coalition ref
input, (b) `Harvest.Transaction.spendVoucher` to plumb the coalition ref
TxIn, (c) `Harvest.Types` to add the new datum types, and (d)
`DevnetEnv` to expose the additional actor keys. New code lives in new
files under `onchain/validators/`, `onchain/lib/harvest/`,
`offchain/src/Harvest/Actions.hs`, and `offchain/test/`. No new
executables.

**Lean-twin discipline**: `offchain/src/Harvest/Actions.hs` is the
twin of `lean/Harvest/Actions.lean`. Each transition has the same
signature shape on both sides (state → action inputs → state or
rejection). Preservation theorems in Lean become QuickCheck properties
over `Harvest.Actions` (added in a later ticket — this plan only
commits to the signature parity, not to the QuickCheck module).

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| Principle IX (three-trie MPF state) is represented as plain sets in a reference UTxO instead of MPF roots | Issue #5 (MPF on-chain) and #8 (MPFS mediation) are scale/usability optimisations, not security properties. Formalising MPF here would delay the first executable full-flow prototype behind a parallel workstream and bury every invariant of §V–§VII under a hash abstraction that adds zero security signal at N ∈ {1,2,3}. | (a) Wait for #5/#8 to land first — rejected because the declared blocker `#8 → #9` was already skipped by #15 along the same path, with the user's explicit approval captured in the spec. (b) Model both MPF and set-based in parallel here — rejected because it doubles the surface under test without evidence the invariants differ. The refinement obligation is recorded for the later tickets. |
